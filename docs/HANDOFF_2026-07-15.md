## Product behavior now implemented

### Background refresh without losing user work

The original implementation changed a React `key` around an entire route every
poll interval. A later version stopped keying the route but still transiently
published an empty agent registry while the registry query refreshed, which made
`RequireAgent` remove the page.

The current implementation:

- keeps the route mounted;
- publishes a refresh revision through `LiveRefreshContext`;
- refreshes only query-backed consumers;
- preserves the last successful agent list during a background registry reload;
- does not clear labeling inputs, approval state, selected tabs, or onboarding
  drafts merely because backend data refreshed;
- wires Compare, Approvals, Activity, Lineage, and Overview into query-level
  refresh rather than page-level remounting.

### Existing-experiment onboarding

The wizard accepts a populated existing MLflow experiment and includes its trace
history when the experiment is not already claimed by another registered agent.
It creates a separate internal reviewer experiment and persists that ID on the
registry entry. Job-result parsing tolerates normal startup noise and extracts the
structured onboarding result rather than reporting a generic network error after
the backend already detected the experiment.

The first step now collects all shared intake fields in one place:

- existing experiment ID or new experiment name;
- agent/cohort name;
- local project directory;
- natural-language requirements.

### Goal catalog, custom goals, and MemAlign

There is now one onboarding path:

1. Connect the experiment and describe the agent.
2. Choose built-in goals and/or turn requirements into custom goals.
3. Review the exact data gates.
4. Register the agent.

For custom requirements:

- the Python requirements engine recommends/routs dimensions;
- exact quantities such as latency, cost, and token use remain deterministic L0
  metrics;
- subjective dimensions become custom trace-based MemAlign judges;
- the user previews the proposed objective, guardrails, scorer types, and target;
- judges are authored only after explicit confirmation;
- the UI explains that custom judges begin untrusted and require at least the
  backend-provided human-label floor plus scored coverage before they can gate an
  optimization claim.

The initial `/api/onboarding/requirements` result now carries each catalog goal's
Python-computed gates, thresholds, `needed` strings, and summary. The client only
filters/unions those backend facts when checkboxes change, so Data Gates updates
immediately without re-deriving threshold policy in TypeScript.

```yaml
  enabled: false

```

The API requires `wait_after_last_change_seconds > 60`; bundle validation alone
did not catch this, so the deployed value is `61`.

One trigger run now:

- skips successfully reviewed traces;
- skips internal/reviewer traces;
- retries traces carrying only `rlm_review_failed`;
- prioritizes never-attempted traces before retries;
- processes bounded batches (`max_reviews=8`, four workers);
- refetches after every batch and continues until no eligible trace remains;
- remembers attempted trace IDs for the duration of the run so trace-store
  propagation delay cannot select one trace twice;
- fails visibly if selected work produces no successful review;
- does not queue another Jobs run while the active run is draining.

### GEPA Optimize dispatcher and local-machine last mile

The selected agent now has a dedicated `/optimize` route. The page dispatches a
bounded Lakeflow Job through AppKit's Jobs plugin, polls only its own status panel,
and preserves the rest of the UI. Dispatch requires an explicit cost confirmation.
One active run is allowed and job queueing is disabled, so costly requests do not
build up behind an existing run.

This uses `gepa.optimize` / Optimize Anything directly, not
`mlflow.genai.optimize_prompts`. The artifact being optimized is a coding-agent skill
body, and the custom adapter evaluates isolated baseline/candidate agent executions
with the existing correctness harness. The held-out split is not passed to GEPA.

A changed candidate becomes actionable only when its final held-out comparison is
strictly positive and correctness holds. The job then:

1. logs `gepa/gepa_candidate.json` to the separate reviewer experiment;
2. inserts one experiment-scoped pending `gepa_prompt` proposal containing the exact
   diff, target, hashes, artifact URI, evidence, and validation argv;
3. prints a schema-validated output marker for the app.

Approving the proposal sets `approved / waiting_for_companion`; hosted compute does
not write a user's machine. The local companion downloads that exact artifact,
checks the project-relative target and seed/candidate hashes, snapshots the original,
rewrites atomically, runs validation, rolls back on failure, and records lineage.
GEPA proposal reads and terminal writes are scoped by `(agent_name, experiment_id)`.

Current execution caveat: the packaged job supports `claude_code`; Codex and custom
coding agents are rejected before optimizer spend until their live adapters are
packaged.

## Known limits and next checks

1. Local OAuth must remain current for trace reads or authenticated browser
   automation. The final Playwright run completed with the current `dais-demo`
   profile.

2. New-agent RLM wake-up depends on a derivable registry `annotations_table`.
   Onboarding reconciles that agent immediately; deploy bootstrap re-adds the full
   registry after DAB resets the trigger. Underivable rows are reported and need an
   operator-supplied trace location rather than a guessed table.

3. No manual catch-up RLM run was started after the trigger deployment. The next
   Claude Code spans-table update will start a drain and include older eligible
   traces visible in the scan. Verify the first arrival-triggered run finishes
   within the two-hour timeout; a large historical backlog may need an explicit
   catch-up run or temporarily larger batch/timeout settings.

4. `max_results=10000` means the runner sees the newest 10,000 traces per scan.
   Normal arrival volumes are far below this. A backlog larger than that needs an
   explicit backfill strategy rather than assuming one drain can discover older
   traces behind the scan window.

5. LoggedModel linkage is automatic only for launchers/wrappers that set the
   registered model ID. Existing traces and external integrations that do not set
   `MLFLOW_ACTIVE_MODEL_ID`/`set_active_model` will still be grouped by the app's
   version tables but will not carry `mlflow.modelId` themselves.

6. The packaged GEPA executor currently supports `claude_code` only. Other coding
   agents remain visible but are refused before optimizer spend until their live
   job adapter is implemented.

7. The repository may contain unrelated local IDE state, worktrees, scratch
   RLM/alignment scripts, `mlflow.db`, `doubly_linked_list/`, and generated analysis
   artifacts. They are not part of this implementation or its PR and must remain
   unstaged unless explicitly desired.

## Important behavior constraints

- RLM must review every eligible subject trace once, retry failed-only markers,
  and never review its own internal traces.
- Reviewer/scorer execution traces belong in the isolated reviewer experiment;
  assessments attach to the subject trace.
- Only companion-produced code/resource changes require human approval.
- Existing populated experiments may be adopted, but separate subject and
  reviewer experiments remain the safe default.
- Human labels drive MemAlign. Do not trust a custom or aligned judge below the
  configured label/agreement/coverage floors.
- Experiment selection must scope all UI and backend reads; never fall back to
  agent name alone where the same agent can have multiple experiments.
- Preserve user work during background refresh and preserve unrelated dirty
  worktree content.
